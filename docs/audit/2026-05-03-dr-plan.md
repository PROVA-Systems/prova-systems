# Audit 22 — Disaster-Recovery-Plan

**Datum:** 03.05.2026 (Sprint S6 Phase 4)
**Auditor:** Claude Code
**Methodik:** pro Subprozessor: Impact + Workaround + RTO + RPO

---

## Recovery-Objectives (Pre-Pilot)

| Metrik | Zielwert | Begründung |
|---|---|---|
| **RTO** (max. Ausfallzeit) | < 4h | SVs arbeiten meist nicht nachts → 4h-Window OK für 99% der Cases |
| **RPO** (max. Daten-Verlust) | < 24h | Täglicher Supabase-Backup |
| **MTBF** (Verfügbarkeits-Ziel) | 99.5%/Monat | 3.6h Downtime/Monat — Vor-Pilot-Phase akzeptabel |

Nach Pilot-Skalierung (50+ SVs): RTO < 1h, RPO < 1h, 99.9% Verfügbarkeit.

## Ausfall-Szenarien

### S1: Supabase ausgefallen

**Impact:** PROVA-App komplett unbenutzbar (DB + Auth + Storage betroffen)

**Häufigkeit:** Supabase-Status-Page zeigt ~99.9% Verfügbarkeit historisch

**Workaround:**
- **Frontend**: Service-Worker-Cache zeigt letzten Stand (Read-Only)
- **Lokale localStorage-Drafts** überleben Outage (Akten-Notizen, Diktate)
- Nach Outage-Ende: Sync via `data-store.js`

**Recovery-Time:** abhängig von Supabase-Outage-Dauer. Typisch 15-60 Min.

**Marcel-Aktion bei Outage:**
1. status.supabase.com prüfen
2. User via Status-Page-Update informieren (geplant: status.prova-systems.de)
3. Supabase-Support kontaktieren (Pro-Plan: 24h-Response)

---

### S2: Netlify ausgefallen

**Impact:** Frontend nicht erreichbar, Functions nicht erreichbar

**Häufigkeit:** Netlify ~99.99% historisch

**Workaround:**
- DNS-Failover zu zweitem Hosting? — komplex, nicht implementiert
- Service-Worker zeigt Offline-Page

**Recovery-Time:** Netlify-Outages typisch < 30 Min

---

### S3: OpenAI ausgefallen

**Impact:** KI-Features nicht verfügbar (Diktat-Transkription, KI-Hilfen)

**Häufigkeit:** OpenAI ~99.5% — gelegentliche Outages

**Workaround:**
- **Fallback dokumentieren:** SV kann Diktat manuell tippen
- **Whisper-Lokal:** Folge-Sprint-Option (whisper.cpp Browser-Build)
- KI-Hilfen sind opt-in — Gutachten-Erstellung ohne KI weiter möglich

**Recovery-Time:** OpenAI-Status

**Marcel-Aktion:** UI-Banner „KI-Service vorübergehend nicht verfügbar — manuelle Eingabe weiter möglich"

---

### S4: PDFMonkey ausgefallen

**Impact:** PDF-Generation funktioniert nicht

**Häufigkeit:** PDFMonkey ~99% historisch

**Workaround:**
- Akten weiter bearbeitbar (Inhalt in DB)
- Nach Recovery: PDF-Generierung nachholbar
- **Long-term**: PDF-Lib (jsPDF, pdfmake) als Backup integrieren — Folge-Sprint

**Recovery-Time:** PDFMonkey-Status

---

### S5: Stripe ausgefallen

**Impact:** Neue Subscriptions können nicht abgeschlossen werden, Webhook-Events stauen sich

**Häufigkeit:** Stripe ~99.99% historisch

**Workaround:**
- Stripe retried Webhook-Events automatisch (3 Tage, Backoff)
- Neue User können ihre Bezahlung nach Recovery abschließen
- **Bestehende User nicht betroffen** (nur Neu-Subscriptions)

**Recovery-Time:** Stripe-Status

---

### S6: SMTP ausgefallen (IONOS)

**Impact:** E-Mail-Versand funktioniert nicht

**Häufigkeit:** IONOS ~99%

**Workaround:**
- Mails werden ins lokale Postfach geschrieben (`smtp-senden.js` Error-Path)
- Marcel kann manuell aus Stripe-Dashboard refunden

**Recovery-Time:** IONOS-Support

---

### S7: GitHub ausgefallen

**Impact:** Deploy-Pipeline broken (Netlify-Auto-Deploy via GitHub-Push)

**Häufigkeit:** GitHub ~99.95%

**Workaround:**
- Netlify CLI direct-Deploy: `netlify deploy --prod`
- Code lokal verfügbar, GitHub ist nur Mirror

**Recovery-Time:** GitHub-Status, max 1-2h historisch

---

### S8: Marcel/Notebook-Ausfall (Bus-Faktor 1)

**Impact:** Niemand kann hotfixen, niemand kann auf Pilot-SVs reagieren

**Wahrscheinlichkeit:** real (Marcel = Solo-Founder)

**Mitigation:**
- **Notfall-Bookmarklet:** `docs/EMERGENCY-BOOKMARKLET.md` (Marcel hat alternativen Login-Pfad)
- **Vertretung TBD:** Marcel sollte mind. 1 Vertrauens-Person mit Lese-Zugang zu Stripe + Supabase-Dashboard einrichten
- **Pilot-SV-Kommunikation:** im Onboarding klar machen dass Solo-Founder-Status = Bus-Faktor 1

**Severity:** organisatorisch HIGH
