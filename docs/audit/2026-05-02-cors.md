### HIGH-1 — Subdomain-Confusion via `startsWith` in `cors-helper.js`

**Datei:** `netlify/functions/lib/cors-helper.js:29`

```js
const allowedOrigin = ALLOWED_ORIGINS.find(o => o && origin.startsWith(o))
  || ALLOWED_ORIGINS[0];
```

**Reproduktion:**
```js
'https://prova-systems.de.attacker.com'.startsWith('https://prova-systems.de')
// → true
```

**Auswirkung:** ein Angreifer mit Domain `prova-systems.de.attacker.com` (registrierbar) erhält `Access-Control-Allow-Origin: https://prova-systems.de.attacker.com` zurück. Bei `Allow-Credentials: true` (cors-helper setzt das immer) kann er XHR-Calls mit User-Cookies/Authorization-Header durchführen.

**CSRF-Voraussetzung:**
1. User ist auf PROVA eingeloggt (Browser hat aktive Session)
2. User besucht Malicious-Site `prova-systems.de.attacker.com`
3. Malicious-Site macht XHR an `app.prova-systems.de/.netlify/functions/<endpoint>`
4. Browser akzeptiert Cross-Origin-Response wegen reflektiertem Origin → Malicious-Site liest die Response

**Limitierung in PROVA:**
- Authorization-Header (Bearer-Token) ist die primäre Auth-Methode — wird von Browser nicht automatisch angehängt bei Cross-Origin
- Wenn Token aber in localStorage liegt (Voll-Cleanup-Sprint Bridge-Layer) und Malicious-Site ihn lesen kann, ist das ein zweites Issue (Audit 11 localStorage)

**Severity:** **HIGH** — Subdomain-Confusion ist klassischer CORS-Bypass-Vektor.
**Phase 1.9 Fix:** ✅ ja

**Fix-Vorschlag:**
```js
function getCorsHeaders(event, methods) {
  const origin = (event && event.headers && event.headers.origin) || '';
  let allowedOrigin = ALLOWED_ORIGINS[0];
  try {
    const url = new URL(origin);
    const isAllowed = ALLOWED_ORIGINS.some(o => {
      try { return new URL(o).origin === url.origin; } catch { return false; }
    });
    if (isAllowed) allowedOrigin = url.origin;
  } catch { /* invalid origin → fallback */ }
  return { /* ... */ };
}
```

### HIGH-2 — Subdomain-Confusion via `includes('prova-systems')` in 4 Functions

**Files:**
- `dsgvo-auskunft.js:17`
- `audit-log.js:14`
- `error-log.js:16`
- `mein-aktivitaetsprotokoll.js:10`

**Pattern:**
```js
'Access-Control-Allow-Origin': (event.headers.origin && (
  event.headers.origin.includes('prova-systems') ||
  event.headers.origin.includes('localhost')
) ? event.headers.origin : 'https://prova-systems.de')
```

**Reproduktion:**
```js
'https://attacker.com/?prova-systems'.includes('prova-systems') // → true
'https://prova-systems.attacker.com'.includes('prova-systems')  // → true
'https://attacker-prova-systems.com'.includes('prova-systems')  // → true
```

**Auswirkung:** noch loser als `startsWith` — jeder Origin der den String „prova-systems" enthält, wird reflektiert. Spezielle Risiko-Endpoints:
- `dsgvo-auskunft` — exportiert User-Daten als Antwort (DSGVO Art. 15)
- `audit-log` — schreibt sicherheitsrelevante Events (kann manipuliert werden)
- `mein-aktivitaetsprotokoll` — User-Aktivitätshistorie

**Severity:** **HIGH**
**Phase 1.9 Fix:** ✅ ja — alle 4 Functions auf `cors-helper.js` umstellen (nach Helper-Fix HIGH-1)

### MEDIUM-1 — `Access-Control-Allow-Origin: '*'` in 5 Functions

**Files:**
- `foto-captioning.js:93, 115, 129` (3 Stellen, alle Error-Paths)
- `ki-proxy.js:152, 356`
- `whisper-diktat.js:200`
- `push-notify.js:467`
- `stripe-portal.js:89`

**Bewertung:** Wildcard-CORS, aber ohne `Allow-Credentials: true`. Browser-Standard:
- Wildcard + Credentials → Browser blockiert
- Wildcard ohne Credentials → erlaubt, Authorization-Header muss explizit gesetzt sein

**Reduzierter Impact in PROVA:**
- alle 5 Functions sind auth-protected (Bearer-Token im Authorization-Header)
- Authorization-Header wird Cross-Origin nicht automatisch übertragen
- Angreifer braucht den Token (aus localStorage stehlen via XSS, oder Phishing)

**Bleibt aber Best-Practice-Verstoß** — Lock-Down auf eigene Origin-Liste verhindert ungewollte Aufrufe.

**Severity:** MEDIUM
**Phase 1.9 Fix:** ✅ ja — alle 5 Functions auf `cors-helper.js` umstellen (auch Error-Paths)

### MEDIUM-2 — `app.prova-systems.de` fehlt in `ALLOWED_ORIGINS`

**Datei:** `netlify/functions/lib/cors-helper.js:12-19`

```js
const ALLOWED_ORIGINS = [
  process.env.URL                            || 'https://prova-systems.de',
  process.env.DEPLOY_PRIME_URL               || '',
  'https://prova-systems.netlify.app',
  'http://localhost:8888',
  'http://localhost:3000',
].filter(Boolean);
```

**Problem:** `app.prova-systems.de` und `admin.prova-systems.de` sind **nicht explizit** in der Liste. Wenn Frontend von App-Subdomain ein Function-Call macht, würde das Allow-List-Match per startsWith greifen weil `https://prova-systems.de` ein Prefix ist (HIGH-1 Bug rettet das ironischerweise — wird mit dem Fix kaputt gehen!).

**Wenn HIGH-1 gefixt wird (exact origin match):** `app.prova-systems.de` würde **nicht matchen** → alle Cross-Subdomain-Calls werden geblockt → **App ist kaputt**.

**Daher Phase 1.9 Fix muss BEIDE Änderungen kombiniert machen:**
1. Origin-Match auf URL-Parsing umstellen
2. ALLOWED_ORIGINS um `app.prova-systems.de` und `admin.prova-systems.de` erweitern

**Severity:** MEDIUM (zwingend mit HIGH-1 gemeinsam fixen)
**Phase 1.9 Fix:** ✅ ja

## Phase-1.9-Fix-Plan

**Single Commit, mehrere Files:**

1. **`lib/cors-helper.js`** rewrite:
   - Origin-Match auf URL-Parsing
   - ALLOWED_ORIGINS erweitern um `https://app.prova-systems.de` und `https://admin.prova-systems.de`

2. **5 Functions mit Wildcard-CORS** auf `cors-helper.js` umstellen:
   - `foto-captioning.js`, `ki-proxy.js`, `whisper-diktat.js`, `push-notify.js`, `stripe-portal.js`

3. **4 Functions mit `includes('prova-systems')`** auf `cors-helper.js` umstellen:
   - `dsgvo-auskunft.js`, `audit-log.js`, `error-log.js`, `mein-aktivitaetsprotokoll.js`

4. **3 Functions mit hardcoded `https://prova-systems.de`** auf `cors-helper.js` umstellen:
   - `normen.js`, `normen-picker.js`, `akte-export.js`

5. **`admin-auth.js`** auf `cors-helper.js` umstellen + ALLOWED_ORIGINS um `admin.prova-systems.de` ergänzen (HIGH-1 Fix deckt das ab)

6. **`termin-reminder.js`, `team-interest.js`, `foto-anlage-pdf.js`, `ki-statistik.js`** mit Pattern 2 (`process.env.URL || 'https://prova-systems.de'`) — optional auf cors-helper umstellen für Konsistenz.

**Aufwand:** ~30 Min für alle Migrations + Helper-Fix.

**Risiko:** Browser-Block bei einer Function die keinen passenden Origin sieht (z.B. wenn Marcel von einem nicht-im-Allow-List-Origin aufruft). Daher: ALLOWED_ORIGINS großzügig genug für alle PROVA-Subdomains.
